# デザイン案: テンプレート同期メカニズム (Template Sync Mechanism)

## 1. 目的

`gemini_agent`（テンプレートリポジトリ）で更新された共通設定（`.github/workflows`, `.gemini/`, `.build/` 等）を、それを利用して作成された各プロジェクトリポジトリ（子リポジトリ）に、プロジェクト固有の設定を破壊することなく安全かつ半自動的に反映させる。

## 2. 背景と課題

- **Git標準機能の限界:** `git merge` ではプロジェクト名（`pyproject.toml`）や `README.md` などの固有部分で毎回コンフリクトが発生する。
- **履歴の汚染:** テンプレート側の全コミット履歴が子リポジトリに混入し、開発履歴が見づらくなる。
- **メンテナンスコスト:** 子リポジトリが増えるたびに、手動でテンプレートの更新を取り込むのは現実的ではない。

## 3. 提案メカニズム: PRベースのディレクトリ同期

### A. 子リポジトリ側からの自発的取得 (Pull型)

各子リポジトリに、定期的にテンプレートの更新をチェックしてプルリクエストを作成する GitHub Action を配置する。

- **動作フロー:**

  1. `schedule` (cron) または `workflow_dispatch` で起動。
  2. テンプレートリポジトリ (`masa-codehub/gemini_agent`) を一時ディレクトリにクローン。
  3. 設定ファイル (`.github/sync-config.yml`) に基づき、同期対象外（`pyproject.toml`, `README.md` 等）を除外して `rsync` で上書き。
  4. 差分がある場合、新しいブランチを作成してプルリクエストを起票。

- **メリット:** テンプレート側で子リポジトリのリストを管理する必要がない。

### B. テンプレート側からの配信 (Push型)

`gemini_agent` でリリースが作成された際、登録されている子リポジトリ群に対して一斉に更新PRを投げる。

- **動作フロー:**

  1. テンプレート側で `release` が `published` になった時に起動。
  2. 対象リポジトリリスト（マトリックス）に対して、現在のテンプレート内容を反映したPRを作成して回る。

- **メリット:** テンプレート更新のタイミングで即座に全プロジェクトへ通知・反映できる。

## 4. 同期ルールの管理

既存の `sync-template-config.yml` の仕組みを拡張し、同期の「ホワイトリスト/ブラックリスト」を明確化する。

```yaml
# 例: 同期設定のイメージ
sync:
  include:
    - .github/workflows/*.yml
    - .gemini/
    - .build/
  exclude:
    - pyproject.toml
    - README.md
    - LICENSE
    - .git/
```

## 5. 推奨される実装ステップ

1. **GitHub Action のテンプレート化:** 子リポジトリにコピーするだけで動作する `sync-template.yml` を `gemini_agent` 内に用意する。
2. **rsyncロジックの共通化:** `script/sync_files.sh` を流用し、上書きコピーとPR作成を行う汎用スクリプトを整備する。
3. **既存ツールとの比較検討:** `Copier` 等のテンプレートエンジンへの移行が必要か、現在のシェルスクリプトベースで十分かを、運用規模に応じて判断する。

## 6. 備考

このデザイン案は、将来的にテンプレートの利用数が増えた際のメンテナンス性を担保するためのロードマップとして保持する。
