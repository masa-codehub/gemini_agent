# Gemini CLI Skill レビューチェックリスト

作成したスキルがベストプラクティスに準拠し、Gemini CLI環境で最適に動作することを確認するためのチェックリストです。

## 1. メタデータと発見可能性 (Metadata & Discovery)

Geminiが適切にスキルを発見し、選択できるか確認します。

- [ ] name はケバブケースか？
  - 例: analyzing-logs (OK), LogAnalyzer (NG), log_helper (NG)
  - 英小文字、数字、ハイフンのみを使用していること。

- [ ] name は具体的で行動指向（動名詞推奨）か？
  - 推奨: deploying-app, checking-security
  - 禁止: tools, helper, utils などの曖昧な名前。
- [ ] description は三人称（Third Person）で記述されているか？
  - 例: "Generates reports..." (OK), "I generate reports..." (NG)

- [ ] description は英語で記述されているか？
  - Gemini CLIのシステムプロンプトとの整合性を高め、発見精度を向上させるため、英語での記述を推奨します。

- [ ] description に「機能」と「トリガー（いつ使うか）」が含まれているか？
  - 機能: 何をするスキルか。
  - トリガー: ユーザーがどのような状況にある時、またはどのようなキーワードを使った時に発動すべきか。

- [ ] description は1024文字以内か？

## 2. 構成とコンテンツの品質 (Structure & Content Quality)

コンテキストウィンドウを無駄遣いせず、効率的かつ明確に情報を伝えているか確認します。

- [ ] SKILL.md は500行未満か？
  - 長すぎる場合は、詳細を references/ ディレクトリに分割しているか。

- [ ] 段階的開示 (Progressive Disclosure) が適用されているか？
  - メインの SKILL.md には概要と主要な手順のみを記述し、API仕様や詳細ルールは別ファイルへのリンクになっているか。

- [ ] 「Geminiは賢い」という前提に立っているか？
  - 一般的な概念（例: 「JSONとは何か」）の説明を省き、このスキル固有の指示に集中しているか。

- [ ] ファイルの参照階層は深すぎないか？-原則として SKILL.md からのリンクは1階層（直接参照）までとし、孫リンク（リンク先からさらにリンク）を避けているか。

- [ ] 用語は一貫しているか？
  - 同じ概念に異なる言葉を使っていないか（例: "URL", "Endpoint", "Route" を混在させず、一つに統一する）。

- [ ] 具体例（Input/Output）が含まれているか？
  - AIに出力形式を指示する場合、説明だけでなく具体的な入出力例（Examples）を提示しているか。

- [ ] 選択肢を与えすぎていないか？
  - 「AかBかCを使えます」ではなく、「Aを使ってください（特別な場合はB）」のようにデフォルトの推奨パスを示しているか。

## 3. スクリプトと技術的実装 (Scripts & Technical Implementation)

"Solve, don't punt"（AIに丸投げせず、道具で解決する）の原則に従っているか確認します。

- [ ] 複雑なロジックはスクリプト化されているか？
  - 複雑な手順をテキストで長々と説明する代わりに、PythonやBashスクリプト（scripts/内）を実行させているか。

- [ ] スクリプトはエラーハンドリングを含んでいるか？
  - ファイルが見つからない場合や引数が不正な場合に、AIが理解できるエラーメッセージを出力するか。
  - AIに「どうすればいいか」を委ねず、スクリプト内でデフォルト動作や回避策を提供しているか。

- [ ] パスはUnixスタイル（/）を使用しているか？
  - Windowsスタイルのバックスラッシュ（\）を使用していないか。

- [ ] 必要な依存ライブラリが明記されているか？
  - ユーザー環境で実行可能か考慮されているか（標準ライブラリの使用推奨）。

- [ ] MCPツール参照は完全修飾名になっているか？
  - MCPツールを使用する場合、ServerName:tool_name の形式（例: GitHub:create_issue）で記述しているか。

4. ワークフローと安全性 (Workflows & Safety)
   - タスクの実行手順が明確で安全か確認します。

- [ ] 複雑なタスクにはチェックリストがあるか？
  - 複数のステップがある場合、AIが進捗を追えるようなチェックリスト形式になっているか。

- [ ] フィードバックループ（検証プロセス）が含まれているか？
  - 「作成 → 検証（スクリプト実行など） → 修正」というサイクルがワークフローに組み込まれているか。
    - 例: 変更を加える前に plan.json を作成・検証する手順があるか。

- [ ] 情報の更新頻度を考慮しているか？
  - すぐに古くなる情報（特定の日付やバージョン依存の強い記述）を避けているか、あるいは「古いパターン」セクションに隔離しているか。

- [ ] 機密情報やハードコードされたパスを含んでいないか？
